## 引言
本文件系统性梳理小说生成系统的"智能体通信机制"，围绕消息路由、协议定义、数据序列化、通信协议、消息传递模式、安全机制、性能优化与监控诊断等方面进行深入解析。本次更新重点介绍了增强的Agent通信机制，包括AgentMesh内存适配器、上下文压缩器和传播器，以及改进的消息传递和状态管理能力。

## 项目结构
该系统以"智能体"为核心，通过统一的通信管理器实现 Agent 之间的消息传递；调度器负责任务生命周期管理与 Agent 分配；具体 Agent 实现各自业务逻辑；同时提供成本追踪、加密存储、监控 API 等配套能力。新增的增强组件提供了更强大的上下文管理和持久化能力。

## 核心组件
- **消息与通信**
  - Message：消息结构，包含消息 ID、发送方、接收方、消息类型、内容、时间戳、优先级与状态
  - AgentCommunicator：消息队列、广播、历史记录与注册管理，支持请求-响应模式
- **上下文管理**
  - ContextCompressor：四层记忆架构（热/温/冷/核心），解决长篇小说上下文膨胀
  - ContextPropagator：将推断约束转化为读者期待的创作指导
  - EnhancedContextManager：智能关键信息提取和动态上下文构建
  - TeamContext：团队共享上下文，支持角色状态追踪和时间线管理
- **持久化存储**
  - AgentMeshMemoryAdapter：SQLite + FTS5 全文搜索的持久化记忆系统
- **调度与任务**
  - BaseAgent：抽象基类，提供消息循环、任务队列与状态管理
  - AgentScheduler：任务生命周期管理、依赖检查、任务分配与完成回调
  - AgentTask：任务结构，包含任务 ID、名称、类型、优先级、依赖、输入输出、超时与回调
- **管理与编排**
  - AgentManager：单例管理器，负责通信器、调度器、Agent 实例的创建与注册
  - AgentDispatcher：在"调度器模式"与"CrewAI风格模式"之间切换，驱动规划与写作阶段
  - CrewManager：直接调用 LLM 完成企划与写作阶段，适合无需 Agent 间消息编排的场景
- **安全与成本**
  - CostTracker：追踪 Token 使用与成本
  - EncryptionService/PublishingService：凭证加密与账号管理
- **监控与日志**
  - MonitoringService/API：系统状态、性能指标、错误分析与健康检查
  - logging_config：统一日志配置与输出

## 架构总览
系统采用"消息驱动 + 任务编排 + 智能上下文管理"的三层架构：
- **消息驱动**：Agent 通过 AgentCommunicator 的队列进行点对点与广播通信，支持任务分配、取消、状态查询等消息类型
- **任务编排**：AgentScheduler 负责任务生命周期与依赖管理，BaseAgent 从队列取出任务并执行，完成后通过消息上报状态
- **智能上下文**：ContextCompressor 和 EnhancedContextManager 提供四层记忆架构，确保上下文大小保持在 ~8K tokens
- **持久化存储**：AgentMeshMemoryAdapter 实现 SQLite + FTS5 的全文搜索，解决内容不连贯问题

### 上下文管理增强

#### 上下文压缩器
采用四层记忆架构，将上下文保持在恒定 ~8K tokens：

- **热记忆 (Hot)**：前 2 章完整摘要 + 前章结尾 500 字
- **温记忆 (Warm)**：第 3-10 章的关键事件列表  
- **冷记忆 (Cold)**：更早章节的卷级摘要
- **核心记忆 (Core)**：世界观 + 角色卡 + 主线（始终携带）

#### 上下文传播器
将推断的约束转化为"读者期待"的创作指导：

- **约束指导模板**：基于上一章的自然推断，提供正向指导而非强制约束

- **近期层**：最近 3 章详细摘要 + 结尾原文
- **历史层**：卷级摘要 + 关键事件索引

### 持久化存储增强

#### AgentMesh 内存适配器
借鉴 AgentMesh 的设计思想，实现 SQLite + FTS5 全文搜索的持久化记忆系统：

- **存储层设计**：SQLite + FTS5 全文搜索，解决内存缓存 30 分钟过期导致的内容不连贯问题
- **表结构设计**：
  - 章节摘要表：key_events、character_changes、plot_progress、foreshadowing、ending_state
  - 角色状态表：last_appearance_chapter、current_location、cultivation_level、emotional_state
  - 小说元数据表：world_setting、characters、plot_outline
  - 伏笔追踪表：foreshadowing_type、importance、expected_resolve_chapter
  - 记忆块表：用于语义搜索的文本块
- **全文搜索**：FTS5 全文索引支持关键词搜索，回退到 LIKE 搜索
- **并发性能**：WAL 模式 + 线程本地连接，提升并发性能

### 调度与任务编排
- **AgentScheduler**
  - 任务状态机：PENDING → ASSIGNED → RUNNING → COMPLETED/FAILED/CANCELLED
  - 依赖检查：仅在依赖任务全部完成时才调度
  - 任务分配：按优先级排序，选择空闲 Agent 分配
  - 完成回调：任务完成后释放 Agent 并触发回调
- **BaseAgent**
  - 消息循环：周期性从队列取消息并分发处理
  - 任务循环：从任务队列取任务并调用子类实现的处理逻辑
  - 状态管理：IDLE/BUSY/ERROR/OFFLINE
- **AgentTask**
  - 输入输出约定：input_data/expected_output
  - 超时与回调：timeout/callback

### 编排模式对比
- **调度器模式（AgentScheduler）**
  - 通过消息驱动任务分配与完成上报，适合复杂依赖与多 Agent 场景

## 性能考虑
- **消息批处理**
  - 可在 Agent 侧聚合多次任务处理结果，减少消息往返次数
- **连接池与并发**
  - LLM 客户端与数据库连接池需合理配置，避免阻塞与抖动
  - AgentMeshMemoryAdapter 使用线程本地连接，提升并发性能
- **任务优先级与调度**
  - 通过 TaskPriority 与依赖检查，避免热点 Agent 过载
- **超时与背压**
  - receive_message 设置合理超时；队列长度与锁粒度控制防止阻塞
- **成本控制**
  - CostTracker 记录 Token 使用，结合提示词长度与温度参数优化
- **上下文优化**
  - ContextCompressor 将上下文大小控制在 ~8K tokens
  - 增强型上下文管理器智能提取关键信息，避免冗余内容
- **存储性能**
  - SQLite WAL 模式 + FTS5 全文索引，提升查询性能
  - 记忆块缓存减少重复计算

## 故障排查指南
- **日志定位**
  - 使用统一日志配置输出到控制台与文件，便于问题复现与审计
- **监控与健康检查**

- **常见问题**
  - Agent 未注册：检查 AgentManager.initialize 与 Agent.start 是否正确调用
  - 任务卡住：确认依赖是否完成、Agent 是否处于 IDLE 状态
  - 成本异常：核对 CostTracker 记录与 LLM 返回 usage 字段一致性
  - 凭证泄露风险：确保使用 EncryptionService 加密存储敏感信息
  - 上下文丢失：检查 ContextCompressor 和 EnhancedContextManager 的配置
  - 存储异常：验证 AgentMeshMemoryAdapter 的数据库连接和权限
- **调试工具**
  - 消息历史查询：使用 get_message_history 获取通信记录
  - 上下文检查：通过 TeamContext.build_enhanced_context 验证上下文完整性
  - 存储验证：使用 AgentMeshMemoryAdapter 的查询接口验证数据一致性

## 结论
该智能体通信机制以"消息驱动 + 任务编排 + 智能上下文管理"为核心，通过新增的 AgentMesh 内存适配器、上下文压缩器、传播器和增强型上下文管理器，实现了从短篇到长篇小说的完整支持。系统既支持细粒度的 Agent 间协作，也兼容直接编排模式，满足不同规模与复杂度的业务需求。配合成本追踪、加密存储与监控 API，以及 SQLite + FTS5 的持久化存储，形成从通信到运维的完整闭环能力，具备优秀的可扩展性与可维护性。
