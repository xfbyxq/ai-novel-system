# Git工作流规范

## 分支管理策略

### 分支命名规范
- **功能分支**: feature/功能名称-issue编号
  - 示例: feature/user-auth-123
- **修复分支**: bugfix/问题描述-issue编号
  - 示例: bugfix/fix-login-error-456
- **热修复分支**: hotfix/问题描述
  - 示例: hotfix/fix-security-vulnerability
- **发布分支**: release/版本号
  - 示例: release/v2.0.0

### 主要分支
- **main**: 生产环境分支，受保护
- **develop**: 开发主分支
- **feature/**: 功能开发分支
- **bugfix/**: Bug修复分支

## 提交规范

### 提交信息格式
```
<类型>(<范围>): <标题>

[可选正文]

[可选脚注]
```

### 类型说明
| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档更新 |
| style | 代码格式调整 |
| refactor | 代码重构 |
| perf | 性能优化 |
| test | 测试相关 |
| build | 构建系统/依赖变更 |
| ci | CI/CD配置变更 |
| chore | 其他变更 |

### 提交示例
```
feat(auth): 添加用户登录功能

- 实现用户名密码登录
- 添加Token认证
- 集成登录验证码

Closes #123
```

## 代码审查规范

### 审查要点
1. **功能正确性**: 代码是否实现预期功能
2. **代码质量**: 命名、注释、复杂度
3. **安全性**: 敏感信息处理、权限控制
4. **性能**: 数据库查询、内存使用
5. **测试**: 是否有足够的测试覆盖

### 审查清单
- [ ] 代码符合项目编码规范
- [ ] 已添加必要的注释
- [ ] 单元测试通过
- [ ] 没有引入安全漏洞
- [ ] 没有性能问题
- [ ] API文档已更新

## 合并策略

### Pull Request流程
1. 从develop分支创建功能分支
2. 在功能分支上开发并提交代码
3. 推动送到远程仓库
4. 创建Pull Request
5. 进行代码审查
6. 审查通过后合并到develop
7. 删除功能分支

### 合并注意事项
- 禁止直接推送到main分支
- 所有PR必须经过审查
- 合并前必须通过CI测试
- 保持提交历史清晰
